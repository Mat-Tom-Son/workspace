# Workspace 0.2.8

Workspace 0.2.8 introduces reviewed Space apps: small interactive tools that the Assistant can build as ordinary files inside a Space, then submit to Workspace for a separate human review and installation decision.

## What is new

- **Space-owned app surfaces.** A reviewed app can contribute an interactive navigator in the Space rail and request normal persistent work tabs. Every surface keeps the identity of its owning Space across tab switches and restarts.
- **A separate restricted runtime.** Visible UI and optional Assistant/background work run in separate ephemeral Electron sandboxes with Node disabled, context isolation enabled, restrictive navigation and networking policy, and sender-bound host bridges.
- **Human review before execution.** Workspace inspects a completed `agent-app.json` package without evaluating its JavaScript, stages the exact reviewed digest, and binds Assistant proposals to the Space and Chat that requested them. A proposal cannot install the package or grant access.
- **Explicit, default-off authority.** Installation grants no network destination, Space file or folder, notification category, stored connection, or background execution. Each power is reviewed and changed separately in Capabilities.
- **Host-owned data and connections.** Apps can use bounded machine-local JSON storage, History-covered Space-file grants, exact public-HTTPS or numeric-loopback destinations, encrypted API-key/bearer/basic connections, and standards-only OAuth PKCE without receiving stored secrets.
- **Background work and Windows notifications.** A separately enabled worker can perform bounded scheduled or manual work. A granted notification selects only static title and body copy reviewed in the manifest; app code cannot provide dynamic copy, actions, or URLs. Notification clicks reopen the exact owning Space and app.
- **Storage refresh without hidden execution.** Active visible app UI receives bounded storage-change hints. Inactive views do not receive queued events and reload durable state when reopened.

## Security and authorization boundary

Restricted apps do not enter Pi's full-trust Extension catalog. Workspace rejects lifecycle scripts, binaries, native build metadata, Pi Extension declarations, unsafe paths, links, excessive files, and oversized packages. Dependency metadata is inert: Workspace never runs npm or resolves dependencies for these packages.

Direct renderer networking, Node and process APIs, arbitrary navigation, downloads, webviews, dialogs, device permissions, and host powers outside an accepted UI, Assistant action, or background invocation are denied. Brokered public HTTPS validates method, origin, redirects, and resolved addresses; numeric loopback is a separate anonymous-only grant. Loopback verifies the reviewed address and port, not which process owns that port.

Space-file writes remain grant-relative, atomic, and covered by targeted History checkpoints. App storage is scoped to Space and app, preserved across reviewed app updates, and deleted when the app or owning Space is removed. Revocation, app stop, suspend, removal, and Workspace shutdown tear down the affected views, workers, active brokered connections, pending storage hints, and outstanding notifications. Stored connection records remain separately visible and revocable in Capabilities until the app or connection is removed.

This is a meaningful reduction in authority, not a claim that arbitrary web code is risk-free. Chromium exploit and CPU or memory denial-of-service risk remain, so Electron updates and the real-Electron restricted-app probe stay release gates. Windows notification text can also appear outside the app window according to system and lock-screen settings.

## Try the checked-in example

The [restricted Connected inbox example](https://github.com/Mat-Tom-Son/workspace/tree/v0.2.8/examples/packages/restricted-connected-inbox) demonstrates the reviewed package, Space rail navigator, persistent tabs, storage and invalidation hints, public-HTTPS and loopback grants, a History-covered export, background work, and static notifications.

For hands-on loopback testing from a source checkout, start the dependency-free demo service:

```powershell
node examples/services/restricted-app-demo-service.mjs
```

Then install `examples/packages/restricted-connected-inbox` through **Capabilities → Apps in this Space → Advanced local install**. Installation should leave every grant off. The example README walks through enabling only the powers needed for each test.

## Upgrade notes

No Space conversion or content migration is required. Updating Workspace preserves registered Spaces, Chats, preferences, Pi state, installed restricted-app receipts and reviewed digests, explicit grants, encrypted connection status, background settings, and machine-local app storage. Removing a restricted app still removes its machine-local storage and connections without deleting ordinary Space files.

Installed Windows builds can check through **Help → Check for Updates…**. Checks do not download automatically; the user chooses when to download and restart. Unpacked development and release-smoke packages intentionally have no updater feed.

## Known limits

- Numeric loopback grants verify the reviewed address and port, not the identity of the process listening there. The included demo service is a development helper that the user starts and stops explicitly; Workspace does not launch or trust it.
- Background schedules run only while Workspace is running. Version 0.2.8 does not include remote subscriptions, arbitrary push adapters, or a verified process-owned Space-service registry.
- Notification copy is static and Windows may suppress presentation through Focus Assist or notification settings. A notification click that arrives exactly while the trusted renderer is starting or recovering can be dropped; normal clicks after the Workspace UI is live reopen the owning app.
- The published installer uses the project's timestamped personal self-signed `CN=Mat-Tom-Son Workspace` identity. It provides artifact continuity but not public Windows trust, so SmartScreen or certificate-chain warnings remain expected.

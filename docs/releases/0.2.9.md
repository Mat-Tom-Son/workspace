# Workspace 0.2.9

Workspace 0.2.9 adds first-class named automations for reviewed Space apps. A Space app can now declare several independently controlled jobs that share one host scheduler across every Space, without turning app code into a full-trust Pi Extension.

## What is new

- **Named automations.** Version-2 `agent-app.json` packages declare up to sixteen jobs with reviewed names, worker handlers, intervals, catch-up behavior, and explicit non-overlap policy. The unreleased single-background-job prototype is intentionally not supported.
- **One scheduler across Spaces.** Workspace coordinates all restricted-app jobs through one machine-wide FIFO queue with two execution slots and same-job non-overlap.
- **Durable cadence and run history.** Enabled schedules retain their cadence across restart and sleep/wake recovery. Manual, scheduled, catch-up, skipped, cancelled, and failed attempts receive bounded machine-local receipts in Capabilities.
- **Independent controls.** Every automation installs off and can be enabled or disabled separately. **Run now** remains available while a schedule is off and never shifts the recurring cadence.
- **Per-job authority.** Each job declares an exact subset of the app's network, Space-file, and notification permissions. Workspace intersects that subset with current grants again at launch.
- **Automation-aware notifications.** Static reviewed Windows notifications require a current category grant, inclusion in the running job's permission subset, and an enabled schedule. A one-off run of a disabled job cannot notify.
- **Clearer app review.** Installation review and Capabilities now show each job's schedule, catch-up behavior, power subset, current grants, next run, last result, and recent receipts.

## Runtime and lifecycle

Automation workers run in the existing hidden Electron sandbox with Node and direct network access denied. The worker exports `handleAutomation(event)` and receives host-authored run identity, automation id, handler, reason, and scheduled time. App updates reset grants, connections, schedules, and run receipts while preserving bounded local app storage.

Workspace cancels active automation work on schedule disable, suspend, update, removal, and shutdown. Schedule teardown is coordinated with durable registry writes so a failed persistence operation cannot silently discard an installed schedule. A one-off manual run does not alter the durable cadence anchor.

The restricted-app scheduler remains outside management protocol v1. The CLI stays read-only; Capabilities and the Space-scoped local API own the authenticated mutation path.

## Try the checked-in example

The [restricted Connected inbox example](https://github.com/Mat-Tom-Son/workspace/tree/v0.2.9/examples/packages/restricted-connected-inbox) now declares a **Refresh inbox** automation. Install it through **Capabilities → Apps in this Space → Advanced local install**, grant only the powers you want to exercise, then enable the automation or choose **Run now**. Its UI reloads the durable result from host-owned storage and its recent receipts remain visible in Capabilities.

## Upgrade notes

This application is still in development and the old restricted-app manifest was used only by the checked-in example. Version 0.2.9 therefore adopts version 2 directly instead of carrying a compatibility layer. Reinstall or update any local experimental app with a version-2 manifest containing an `automations` array; use an empty array for apps without scheduled work.

Registered Spaces, Chats, preferences, Pi state, and ordinary Space files are unaffected. Restricted-app storage remains machine-local and survives a reviewed app update, while the new digest must receive fresh grants, connections, and automation enablement.

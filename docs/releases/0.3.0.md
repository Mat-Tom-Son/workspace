# Workspace 0.3.0

Workspace 0.3.0 establishes the local-first App-platform foundation beneath the
existing reviewed Space-app experience. It keeps a Space as an ordinary
folder-backed context while giving an optional App Project, its Features, data,
connections, automations, releases, instances, tenants, and actors precise and
separate identities.

This is deliberately a foundation release. Existing version-2 Space apps keep
their current UI and package contract. Release publication, release-backed App
Instance installation, accounts, hosted deployment, and an App Store are not
new desktop features in this version.

## What is new

- **A settled App product model.** Space, App Project, Feature, immutable App
  Release, Development Instance, release-backed App Instance, Tenant, and
  Principal now have one documented ownership and lifecycle model. Publication
  remains separate from folder sync, installation, grants, connections, and
  automation enablement.
- **Explicit local runtime identity.** The restricted-app registry now records a
  local Project and Development Instance plus stable Feature Installation and
  Data Namespace identities. Local operation uses an account-free Tenant and
  explicit human and service Principals.
- **Seven independently fenced authority domains.** Runtime Instance, Feature
  Installation, grant, connection, job, Principal, and data generations advance
  independently. Hosts revalidate the exact captured authority immediately
  before network, OAuth, storage, and Space-file effects.
- **Stronger audit lineage.** New automation receipts record the accepting
  Tenant, runtime, installation, canonical revision, data namespace, effective
  Principal, authority, occurrence, and attempt without storing app payloads,
  file contents, request bodies, or credentials. Acceptance is durable before
  worker execution and terminalizes by run id even after update or removal.
  A restart with an accepted but unfinished run records an explicit
  `interrupted`/`expired` result whose external-effect completion is unknown,
  rather than inventing a success, failure, or cancellation.
  Read and write registry bounds now match, and admission reserves room for
  worst-case terminal receipts before a worker starts.
  Older receipts remain visible and are labeled `legacy-unverified` rather than
  receiving invented facts.
- **Crash-safe cleanup.** Update, app removal, and Space removal commit durable,
  idempotent cleanup obligations with the authority change. Interrupted secret,
  storage, and staged-package cleanup is retried at startup and before later
  mutations; stale bytes cannot make an old installation live again.
- **Portable immutable artifacts and Releases.** A specified
  `workspace-artifact-v1` framing, RFC 8785 declaration canonicalization,
  language-neutral vectors, an implementation-independent verifier, and a
  strict offline multi-Feature Release assembler/verifier make content identity
  testable without a server.
- **Deterministic update planning.** The local App Instance planner binds the
  active and target Release to one Project, resolves one exact directed schema
  migration path, preserves unchanged Feature continuity, requires durable
  caller-owned allocation for additions, and exposes a single atomic activation
  point. It is platform code in 0.3.0, not yet a desktop update UI.
- **Private hosted contract evidence.** An executable semantic core proves
  authenticated Project/cloud-Project and Tenant separation, private review and
  publication, deployment, default-off grants, instance-owned connections,
  leased jobs, effect-time revocation, role/principal/instance data, compatible
  update, export, delete, purge, and restart recovery. It uses injected durable
  repository, scheduler, vault, and egress interfaces and is not a deployed
  Workspace cloud service.
- **Two validation fixtures.** The restricted Connected inbox continues to
  exercise the coding-oriented Space-app path. A community-garden fixture
  exercises shared role data, private preferences, reminders, export, and
  deletion without assuming an IDE or cloud account for the local Space.

## Security and authority changes

Visible and worker hosts capture a launch generation and fail closed after stop,
unmount, crash, close, authority change, or installation replacement. Network
authority is checked before DNS and before every request or redirect. OAuth is
checked before browser and provider effects, token injection, encrypted-store
access, and atomic encrypted-store commit. Storage and Space-file writes check
again after temporary-file sync and immediately before the atomic commit.

Restricted-package review now reads only the exact regular files and sizes
captured during enumeration, through bounded handles with identity checks
before and after each read. The manifest, totals, package digest, and portable
artifact digest come from one byte snapshot; a concurrent replacement is
rejected rather than becoming unreviewed input or an oversized allocation.

Connections now bind the exact Tenant, Runtime Instance, Feature Installation,
canonical Feature Revision, declaration, destination, origin, and current
Runtime Instance owner. Principal-owned connection consent and unattended
delegation remain future product work. App JSON storage is owned by the Tenant and Data Namespace,
while ordinary Space files remain ordinary user files and stay outside app
storage.

The private hosted core is security contract evidence only. A production service
would still require real database/CAS, scheduler and lease workers, secret vault,
SSRF-resistant egress, authenticated transport, deployment, monitoring,
retention, regional, support-access, and UI implementations before it could host
user data.

## Upgrade notes

Version-2 `agent-app.json` packages remain supported without a package rewrite.
Installed apps, reviewed package digests, explicit grants, automation settings,
historical receipts, and app storage migrate into the stronger local identities.
Existing Space/app storage is adopted once into the matching local Tenant and
Data Namespace.

Connections written by Workspace 0.2.9 or earlier cannot prove the new runtime,
installation, revision, and owner identity. They intentionally all appear
disconnected after upgrade and the encrypted legacy store remains untouched
until the first explicit reconnect. That reconnect creates the stronger store
and discards every ambiguous legacy binding; Workspace does not silently
transfer their authority. Use provider-side revocation as well when an old
credential should no longer be valid at its provider.

Registered Spaces, Chats, preferences, Pi state, Skills, Extensions, History,
Library materials, and ordinary Space files are unaffected. Removing a Space now
also revokes its local App Project and Development Instance scaffold even when
its last Space app was already removed.

## Known limits

- The desktop still presents the version-2 Space-app install and Capabilities
  surface; App Release publication and App Instance installation have no product
  UI yet.
- The hosted semantic core has no production service or public endpoint.
- The first hosted update slice accepts only an unchanged Feature set with
  compatible data schemas and collection policies. It rejects migrations and
  Feature-set changes instead of applying them partially.
- Restricted apps remain a separate reviewed web lane and never become native Pi
  Extensions or inherit ambient Pi, Library, Chat, or Space-folder authority.
- Google Drive remains a Drive-for-desktop folder workflow rather than native
  cloud integration. General model-provider OAuth is still future work.
- Public releases may remain unsigned or self-signed until a publicly trusted
  signing identity is configured, so Windows may warn about the publisher.

## Release verification

The release gates include TypeScript checks, the full test suite, dependency
audit, cross-implementation digest conformance, migration and stale-authority
tests, native Pi preflight, the real-Electron restricted-app probe, renderer
production build, packaged-layout checks, updater metadata verification, and
tagged cloud publication from the pushed `main` commit.

On July 16, 2026, the guarded Apple silicon release lane Developer ID-signed,
notarized, stapled, verified, and published all eight macOS 0.3.0 artifacts to
the separate public Mac feed. An installed 0.2.11 app then downloaded a ZIP
matching the published SHA-256
`d8bb801498c2823f54b10340395b0ee2428b3154c33c0b4b49658106c269bcea`,
replaced itself, and relaunched as 0.3.0. The registered Space and capability
projection were preserved; native version/update checks, Quit, Command-Q,
system termination, and close/reopen behavior passed without a SecurityAgent
or Keychain dialog.

# Workspace 0.4.0

Workspace 0.4.0 turns the local App-platform foundation into a complete desktop
workflow. A reviewed Space app is now clearly a **Local preview**; App Studio can
close those exact bytes into an immutable Release, publish it for local use, and
install it into a chosen Space as a separate App Instance.

Everything in this release remains local first. App Projects, Releases, App
Instances, data, grants, connections, automations, and receipts stay on this
computer. “Publish” does not upload an App, deploy a hosted service, synchronize
a Space, create an account, or submit anything to an App Store.

## What is new

- **A Space-bound App Studio.** Capabilities opens a persistent App Studio work
  tab without adding another top-level rail destination. It separates Project
  setup, reviewed previews, Releases, installations, prepared operations, and
  retained data.
- **Explicit App presentation.** One source Space can declare a machine-local
  App Project title, description, and icon. Existing previews can seed sensible
  defaults, but Workspace does not add a portable App Project file or treat a
  copied folder id as ownership.
- **Immutable Release format v2.** Preparing a Release snapshots every current
  reviewed Development Feature. The digest covers the App presentation, exact
  Feature packages and declarations, dependency inventory, build provenance,
  and inspection evidence. Canonical envelopes live in a bounded,
  content-addressed, crash-recovered local store.
- **Preparation and publication stay separate.** A prepared Release is not
   installable. Local publication rechecks that every preview still matches the
   prepared source stamps and records a separate receipt. If a preview changed,
   Workspace requires a new Release instead of publishing stale content.
- **Bounded, explicit Release cleanup.** The local Release store enforces a
  four-GiB aggregate quota. App Studio can delete an unused prepared or
  published Release, but never one required by an active Instance, either side
  of a prepared operation, or retained-data lineage. Registry deletion commits
  before safe physical pruning, and interrupted cleanup is retried.
- **Release-backed local App Instances.** Installation is prepared and then
  activated in a chosen registered Space. The durable operation reserves the
  Runtime Instance, Feature Installation, and Data Namespace identities before
  activation; activation re-verifies and stages the exact closure before making
  it live. Preview data and authority never transfer, and every network, file,
  notification, connection, and automation power starts disabled.
- **Deterministic update and rollback.** App Studio can prepare a transition to
  any other published Release from the same Project, including an older Release.
  Activation recomputes the persisted plan and refuses stale state. Exact
  unchanged Features may keep eligible grants, connections, jobs, and data;
  changed Features keep their installation/data lineage but reset powers and
  current-revision run state.
- **Explicit uninstall data handling.** Whole-App uninstall requires **retain**
  or **purge**. Retained Data Namespaces have no live installation authority and
  appear in App Studio for a later explicit purge. Uninstall never deletes App
  source or separately selected ordinary Space files.
- **Restart-safe lifecycle journals.** Prepared install and update operations,
  Release states, administrative receipts, retained-data records, and cleanup
  obligations are durable. Interrupted cleanup is retried without making stale
  authority live again.
- **Crash-safe Space removal.** Workspace records the removal intent before
  revoking App authority or deleting content. Interrupted removals stay hidden
  and untrusted until recovery finishes. A whole-Space runtime fence removes
  broker and scheduler authority immediately. Managed-folder cleanup atomically
  claims and revalidates the exact approved directory identity under a random
  same-filesystem path, so a replacement folder or junction at the registered
  path is never recursively deleted by a retry.

## Placement and lifecycle rules

Version 0.4 intentionally starts with one local App Instance per `(App Project,
target Space)`. A target Space cannot already contain a Development preview or
installed Release Feature with the same Feature id. Installing into the source
Space therefore requires removing its conflicting preview first.

The App Instance is attached to its target Space for navigation and explicit
file-grant eligibility, but its executable bytes, data, grants, connections,
schedules, journals, and receipts stay in Workspace application data. Workspace
blocks removing either the Project's source Space or an active App Instance's
target Space until the App Instance is uninstalled with an explicit data choice.
Retained data continues to block the source until explicit purge, but does not
bind the former target. Removing an obligation-free source clears the local App
Project and Release lineage, then safely reconciles unreferenced Release
objects. Transient cleanup failure retries before later App mutations and at
startup. Removing a target cancels unactivated operations aimed there.

## Security and authority

Release preparation does not execute package JavaScript, resolve dependencies,
or run build hooks. Format-v2 objects are verified before storage and again
before installation or update staging. Content-addressed directories reject
links, unsafe entries, non-canonical objects, digest mismatches, concurrent file
identity changes, and configured size-limit violations.

Install and update preparation does not itself change live authority. Activation
checks affected Spaces for concurrent Assistant/capability work, verifies the
published Release and durable plan, stops predecessor hosts before the authority
fence, and commits the active Release transition once. Changed code cannot keep
old grants, connections, or jobs. The local runtime rejects Releases that carry
a data schema or migration rather than executing a migration it cannot yet
enforce safely.

## Upgrade notes

The restricted-app registry migrates from schema version 3 to version 4.
Existing reviewed apps remain Development previews with their Feature
Installation, Data Namespace, grants, automation state, and run lineage intact.
Their App Project presentation defaults from the installed Feature where
available. Migration does not fabricate Releases, App Instances, prepared
operations, retained data, or administrative receipts.

Version-2 `agent-app.json` packages and the sandbox bridge remain compatible.
The behavior change is product placement: the existing Chat-bound and advanced
local preview paths are Development previews, while App Studio is the only path
that creates or removes a release-backed App Instance.

Registered Spaces, Chats, ordinary files, Library materials, History, Pi state,
Skills, Extensions, provider settings, and existing encrypted restricted-app
connection records are otherwise preserved by the in-place update.

## Known limits

- App Project identity and presentation are machine-local. There is no portable
  Project import/relink/collision protocol yet.
- Local publication is not cloud publication. Workspace 0.4 has no hosted
  runtime, account service, upload, cross-device sync, public listing, or App
  Store.
- The local restricted runtime accepts only release Features with no data schema
  or migrations. Migration execution, retained-data adoption, and App-data
  export remain future reviewed management operations.
- One `(projectId, target Space)` instance and the target-Space Feature-id
  collision rule are deliberate first-release constraints.
- Restricted Apps remain a separate reviewed web lane. They do not become
  native Pi Extensions or inherit ambient Pi, Library, Chat, or Space-folder
  authority.

## Release verification posture

This document describes the 0.4.0 release candidate; it does not by itself
certify that a public tag or installer has been published. Before release, the
exact pushed `main` commit must pass TypeScript checks, the full test suite,
dependency audit, release/digest conformance, registry migration and
stale-authority tests, production renderer and Electron compilation, native Pi
preflight, the real-Electron restricted-app probe, packaged-layout smoke, and
Windows installer/updater verification. The clean `v0.4.0` tag must point to
that green commit, and the installer, blockmap, `latest.yml`, and checksums must
come from the same verified build.

The macOS production feed is a separate guarded workstation lane. A Windows
0.4.0 release does not imply that matching signed/notarized Mac artifacts have
already been published.
